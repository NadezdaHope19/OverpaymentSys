# Техническое задание "Система учёта переплат"

## 1. Цели и задачи системы
 
1.1. Цель разработки
 
Автоматизация учёта переплат получателей выплат, контроля погашений, отслеживания движения судебных дел, аналитики.
 
1.2. Основные задачи
 
* Централизованный учёт должников и переплат
* Фиксация всех платежей с автоматическим пересчётом остатка долга
* Контроль движения судебных дел (от подачи иска до окончания исполнительного производства)
* Разграничение доступа по ролям (администратор, юрист, руководитель, оператор)
* Формирование отчётности по должникам, переплатам и судебным делам
 
 
## 2. Функциональные требования
 
2.1. Управление справочниками
 
2.1.1. Источники переплаты (sources)
 
* Добавление, редактирование, удаление (мягкое/жёсткое) записей
* Поля: код, наименование, признак активности
 
2.1.2. Причины переплаты (reasons)
 
* Добавление, редактирование, удаление записей
* Поля: код, наименование, описание, признак активности
* Примеры: «Счётная ошибка», «Ошибка ПО», «Снятие после смерти»
 
2.1.3. Роли пользователей (roles)
 
* Предустановленные роли: ADMIN, LEADER, LAWYER, OPERATOR
* Возможность добавления новых ролей
* Назначение разрешений ролям
 
2.1.4. Разрешения (permissions)
 
* Фиксированный набор: CREATE, READ, UPDATE, DELETE для каждого ресурса
 
2.2. Управление должниками (debtors)
 
2.2.1. Карточка должника содержит
 
| Поле | Тип | Обязательность |
| --- | --- | --- |
| ФИО | STRING | Да |
| СНИЛС | STRING, маска ввода 999-999-999 99 | Да |
| Место работы | STRING | Нет |
| Сумма переплаты | DECIMAL(10,2) | Да |
| Источник переплаты | ссылка на справочник | Да |
| Причина переплаты | ссылка на справочник | Да |
| Период переплаты (начало) | DATE | Нет |
| Период переплаты (конец) | DATE | Нет |
| Дата постановки на учёт | DATE | Да |
| Дата направления уведомления | DATE | Нет |
| Дата направления в юриический отдел | DATE | Нет |
| Остаток переплаты | DECIMAL(10,2) (вычисляемое) | Да |
 
2.2.2. Бизнес-логика
 
* Сумма переплаты > 0 → переплата активна
* Сумма переплаты < 0 → излишек (вернуть должнику)
* overpayment_balance = ABS(debt_amount) если debt_amount < 0, иначе 0
* При изменении суммы долга автоматически пересчитывается остаток переплаты
 
2.2.3. Операции с должниками
 
* Создание новой записи
* Просмотр списка (с фильтрацией и поиском)
* Просмотр карточки должника
* Редактирование (с ограничениями по ролям)
* Удаление (мягкое — скрытие, или жёсткое — только ADMIN)
 
2.3. Управление платежами (payments)
 
2.3.1. Поля платежа
 
| Поле | Тип | Обязательность |
| --- | --- | --- |
| Должник | ссылка на debtors | Да |
| Дата платежа | DATE | Да |
| Сумма | DECIMAL(10,2) > 0 | Да |
| Источник переплаты | ссылка на sourses (из введенных в debtors по должнику) | Да |
 
2.3.2. Бизнес-логика (автоматически)
 
При добавлении/изменении/удалении платежа:
 
1. Пересчитывается текущий остаток переплаты
2. Обновляется debtors.debt_amount:
   * остаток переплаты = остаток переплаты - сумма_платежа
 
2.3.3. Операции с платежами
 
* Добавление платежа
* Просмотр истории платежей по должнику
* Редактирование (только для OPERATOR, LEADER и ADMIN)
* Удаление (только ADMIN)
 
2.4. Управление судебными делами (court_case)
 
2.4.1. Поля судебного дела
 
| Поле | Тип | Обязательность |
| --- | --- | --- |
| Должник | ссылка на debtors | Да |
| Дата направления искового заявления | DATE | Нет |
| Наименование суда | STRING | Нет |
| Дата принятого решения | DATE | Нет |
| Решение | STRING | Нет |
| Исполнительный лист | STRING | Нет |
| Дата поступления исполнительного листа | DATE | Нет |
| Дата окончания исполнительного производства | DATE | Нет |
 
2.4.2. Операции с судебными делами
 
* Создание дела по должнику
* Просмотр карточки дела
* Редактирование (только LEADER, LAWYER, ADMIN)
* Удаление (только ADMIN)

2.5.3. Управление пользователями (только ADMIN)

## 3. Нефункциональные требования
 
3.1. Производительность
 
* Время отклика интерфейса: не более 2 секунд при стандартных операциях
* Поддержка одновременной работы: не менее 20 пользователей
 
3.2. Надёжность
 
* Автоматический пересчёт остатка долга при любых изменениях платежей (через триггеры БД)
* Защита от конкурентных изменений (блокировки строк при обновлении)
 
3.3. Безопасность
 
* Разграничение доступа на уровне приложения (не только интерфейса)
* Защита от SQL-инъекций (параметризованные запросы/ORM)
* Парольная политика: длина ≥ 8 символов
 
3.4. Масштабируемость
 
* Возможность добавления новых ролей и разрешений без изменения кода
* Поддержка добавления новых полей в таблицы
 
3.5. Удобство использования
 
* Поиск и фильтрация по ФИО, СНИЛС, периоду, сумме долга/переплаты
* Сортировка списков по любому полю
* Вывод остатка долга и переплаты в наглядном виде (отрицательные суммы — красным)
 
 
## 4. Состав и структура БД
 
4.1. Перечень таблиц
 
| Таблица | Назначение |
| --- | --- |
| debtors | Должники и переплаты |
| sources | Справочник источников переплаты |
| reasons | Справочник причин переплаты |
| payments | Платежи |
| court_case | Движение судебных дел |
| users | Пользователи |
| roles | Роли |
| user_roles | Связь пользователей с ролями |
| permissions | Разрешения |
| role_permissions | Связь ролей с разрешениями |
 
4.2. Связи между таблицами
 
* debtors → overpayment_sources (N:1)
* debtors → overpayment_reasons (N:1)
* debtors ← payments (1:N)
* debtors ← court_case (1:N)
* payments → overpayment_sources (N:1)
* users ← user_roles (1:N)
* roles ← user_roles (1:N)
* roles ← role_permissions (1:N)
* permissions ← role_permissions (1:N)
 
4.3. Индексы (рекомендуемые)
 
* debtors.full_name (для поиска)
* debtors.snils
* payments.debtor_id
* court_case.debtor_id
* user_roles.user_id, user_roles.role_id
 
 
## 5. Требования к отчётности
 
5.1. Стандартные отчёты
 
| Отчёт | Содержание |
| --- | --- |
| Статистика переплат на текущую дату | сгруппированные по источникам переплаты (количество, сумма) |
| Статистика переплат за период | сгруппированные по источникам (количество, сумма, количество погашенных, сумма погашеных, процент) |
| Реестр платежей | для формирования указать период (можно отфильтровать по должнику и периоду) |
| Судебные дела | Должник, дата иска, решение, исполнительный лист |
| Оборотно-сальдовая ведомость | Начальный долг, платежи, текущий долг/переплата |
 
5.2. Экспорт
 
* Форматы: XLSX, CSV, PDF
* Доступ: OPERATOR, LAWYER, LEADER, ADMIN
